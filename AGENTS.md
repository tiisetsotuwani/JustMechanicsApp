# Instructions for AI Coding Agents (Codex / Devin)

## Before Making Any Changes
1. Read `guidelines/Guidelines.md` for project rules and architecture
2. Read `src/shared/types.ts` for all type definitions
3. Run `pnpm test` to verify existing tests pass
4. Check `BACKEND_API.md` for existing API endpoints

## When Adding a New Frontend Component
1. Define props interface in the component file, importing shared types from `src/shared/types.ts`
2. Create the component in `src/app/components/`
3. Add the screen name to the `Screen` type in `src/shared/types.ts`
4. Add routing in `src/app/App.tsx` `renderScreen()` switch
5. Write a test in `src/__tests__/components/ComponentName.test.tsx`
6. Run `pnpm test` to verify

## When Adding a New Backend Route
1. Add types to `src/shared/types.ts` if new entities are involved
2. Create or modify the route handler file in `supabase/functions/server/`
3. Register the route in `supabase/functions/server/index.tsx`
4. Add the API client method in `src/utils/api.ts`
5. Update `BACKEND_API.md` with the new endpoint documentation
6. Write a test in `src/__tests__/api/` for the API client method

## When Fixing a Bug
1. Write a failing test that reproduces the bug FIRST
2. Fix the bug
3. Verify the test passes
4. Check if the same pattern exists elsewhere in the codebase

## Type Safety Rules
- NEVER use `any` type. Use proper interfaces from `src/shared/types.ts`
- If a type doesn't exist yet, add it to `src/shared/types.ts` before using it
- The booking status values are: `'pending' | 'assigned' | 'en_route' | 'arrived' | 'in_progress' | 'completed' | 'cancelled' | 'disputed'`
- The user types are: `'customer' | 'provider' | 'admin'`
- The provider status values are: `'pending_review' | 'approved' | 'suspended' | 'rejected'`

## KV Store Key Patterns
When storing data in the KV store, follow these key patterns:
- User profile: `user:{userId}`
- Customer bookings list: `customer:bookings:{userId}`
- Provider bookings list: `provider:bookings:{userId}`
- Individual booking: `booking:{timestamp}:{userId}`
- Pending bookings queue: `bookings:pending`
- Addresses: `addresses:{userId}`
- Vehicles: `vehicles:{userId}`
- Provider availability: `provider:availability:{userId}`
- Provider services: `provider:services:{userId}`
- Audit log entry: `audit:{timestamp}:{userId}`

## Common Mistakes to Avoid
- Don't hardcode mock data in components — always call the API
- Don't define types inline — import from shared types
- Don't forget to handle loading and error states in UI
- Don't use `fetch` directly in components — use `src/utils/api.ts`
- Don't forget to register new routes in `index.tsx`
- Don't modify `kv_store.tsx` — it's autogenerated
- Don't expose the Supabase anon key or service role key in frontend code (the anon key in `utils/supabase/info.tsx` is the only exception — it's a public key)
