# Instructions for AI Coding Agents (Codex / Devin)

## Before Making Any Changes
1. Read `guidelines/Guidelines.md` for project rules and architecture.
2. Read `src/shared/types.ts` for all shared type definitions.
3. Run `corepack pnpm test` to verify the current test suite passes.
4. Check `BACKEND_API.md` for existing API endpoints and route conventions.

## When Adding a New Frontend Component
1. Define the props interface in the component file and import shared types from `src/shared/types.ts` when applicable.
2. Create the component in `src/app/components/`.
3. Add the screen name to the `Screen` type in `src/shared/types.ts` if it is a routed screen.
4. Add routing in `src/app/App.tsx` inside the `renderScreen()` switch if needed.
5. Write or update a test in `src/__tests__/components/`.
6. Run `corepack pnpm test` to verify.

## When Adding a New Backend Route
1. Add types to `src/shared/types.ts` if new entities are involved.
2. Create or modify the route handler file in `supabase/functions/server/`.
3. Register the route in `supabase/functions/server/index.tsx`.
4. Add or update the API client method in `src/utils/api.ts`.
5. Update `BACKEND_API.md` with the new endpoint documentation.
6. Write or update a test in `src/__tests__/api/`.

## When Fixing a Bug
1. Prefer writing a failing test that reproduces the bug first.
2. Fix the bug.
3. Verify the relevant test passes.
4. Check whether the same pattern exists elsewhere in the codebase.

## Type Safety Rules
- Never use `any`. Use proper interfaces from `src/shared/types.ts`.
- If a type does not exist yet, add it to `src/shared/types.ts` before using it.
- Booking status values are `'pending' | 'assigned' | 'en_route' | 'arrived' | 'in_progress' | 'completed' | 'cancelled' | 'disputed'`.
- User types are `'customer' | 'provider' | 'admin'`.
- Provider status values are `'pending_review' | 'approved' | 'suspended' | 'rejected'`.

## API And Routing Rules
- Frontend components should use `src/utils/api.ts` instead of calling `fetch` directly.
- Messaging flows should use the existing `/chat/*` routes through `api.chat.*`, not ad hoc `/messages/*` paths.
- Keep auth token management centralized through `api.setAuthToken()` and session restoration in `src/app/App.tsx`.
- Before adding a new endpoint, verify that a matching route does not already exist in `BACKEND_API.md` and `supabase/functions/server/index.tsx`.

## KV Store Key Patterns
When storing data in the KV store, follow these key patterns:
- User profile: `user:{userId}`
- Customer bookings list: `customer:bookings:{userId}`
- Provider bookings list: `provider:bookings:{userId}`
- Individual booking: `booking:{timestamp}:{userId}`
- Pending bookings queue: `bookings:pending`
- Addresses: `addresses:{userId}`
- Vehicles: `vehicles:{userId}`
- Provider availability: `provider:availability:{userId}`
- Provider services: `provider:services:{userId}`
- Audit log entry: `audit:{timestamp}:{userId}`

## Common Mistakes To Avoid
- Do not hardcode mock data in production components. Use the API.
- Do not define shared business types inline. Import them from `src/shared/types.ts`.
- Do not forget loading and error states in async UI flows.
- Do not use `fetch` directly in components. Use `src/utils/api.ts`.
- Do not forget to register new backend routes in `supabase/functions/server/index.tsx`.
- Do not modify `supabase/functions/server/kv_store.tsx`. It is autogenerated.
- Do not expose the Supabase anon key or service role key in frontend code. The public anon key in `utils/supabase/info.tsx` is the only exception.

## Current Repo Notes
- `corepack pnpm test` is the expected test command in this repo.
- Some legacy files still contain direct `fetch`, demo fallbacks, or local mock state. Treat those as cleanup targets, not as patterns to copy.
- If you touch auth, messaging, notifications, tracking, or payments, verify the component behavior matches `src/utils/api.ts` and the backend routes documented in `BACKEND_API.md`.
